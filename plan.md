# 漫画翻译项目 — 技术方案

---

## 技术选型总览

| 管线阶段 | 模块 | 选定方案 |
|:--:|------|------|
| 1 | 文本检测 | YOLOv8-OBB（主力）+ DBNet（辅助） |
| 2 | 图像增强 | 三级级联增强：CLAHE+锐化 → SVTR直识 → Real-ESRGAN fallback |
| 3 | OCR识别 | PP-OCRv5 + SVTR |
| 4 | 机器翻译 | **CADGT** 五阶段翻译管线（Qwen2.5-VL-2B 底层驱动）+ 两阶段混合调度 |
| 5 | 图像修复 | LaMa（Large Mask Inpainting） |
| 6 | 译文渲染 | **SAAT** 风格感知自适应排版渲染：风格检测→空间计算→二分搜索自适应排版→分层风格渲染→双输出 |
| 7 | GUI交互 | QT6 PySide6 |

## 创新点总述

### 创新一：文本检测 — YOLOv8-OBB + DBNet 双管线检测

**创新命名**：旋转框-分割双模检测（Oriented & Segmentation Dual-Path Detection）

**核心思路**：单一检测器无法同时兼顾竖排文本（旋转框优势）和任意形状文本（分割优势），采取双管线并行、融合输出的策略。

| 检测通路 | 擅长场景 | 技术手段 |
|------|------|------|
| YOLOv8-OBB | 竖直文本、倾斜气泡文本 | 旋转矩形框 + 多角度数据增强 |
| DBNet | 弧形文字、严重变形文字 | 文本区域分割 + 可微分二值化 |

**创新价值**：现有漫画翻译工具（BallonTranslator、manga-image-translator）均使用单一检测器，对竖向文本和弧形文本的兼顾不足。双管线互补填补了这一空白，且两个模型均可轻量部署。

---

### 创新二：图像增强 — 三级级联自适应增强

**创新命名**：三级自适应级联增强（Three-Tier Adaptive Cascade Enhancement, TACE）

**核心思路**：不对全图做超分，而是对文本小区域按需逐级增强，只在必要时调用重武器。

```
级别1：传统CV增强（CLAHE对比度 + Unsharp Mask锐化）
         ↓  置信度足够？→ 直接交付OCR
级别2：SVTR原生抗模糊识别（训练时已做模糊增强）
         ↓  置信度不足？
级别3：Real-ESRGAN文本区域定向超分 → 重试OCR
```

**创新价值**：传统方案要么不做增强（模糊时OCR失败），要么全图超分（几秒延迟且产生伪影）。TACE 将模糊处理的耗时压缩到仅文本框级别（通常只占画面5-10%面积），对清晰文本零额外开销，对模糊文本精准增强。

---

### 创新三：OCR识别 — PP-OCRv5 + SVTR 漫画微调

**创新命名**：SVTR多语种漫画适配（SVTR Comic-Adaptive Multilingual Recognition）

**核心思路**：以 PP-OCRv5 系统框架为基础，将识别骨干从 CRNN 升级为 SVTR（Single Visual Text Recognition），并针对中日英漫画文本进行专属微调。

**技术升级要点**：
- CNN+RNN串行架构 → 全Transformer并行架构，推理速度提升30%+
- 局部感受野 → 全局自注意力，对变形/艺术字体鲁棒性更强
- 利用内置方向分类器实现竖排文本自动校正

**创新价值**：将文本识别领域最新的 Transformer 架构落地到漫画这个细粒度场景，通过漫画语料微调弥补通用 OCR 在艺术字体、手写拟声词上的短板。

---

### 创新四：机器翻译 — CADGT 五阶段翻译管线（核心创新）

**创新命名**：角色感知对话图翻译（Character-Aware Dialogue Graph Translation, CADGT）

**核心思路**：现有所有漫画翻译工具逐气泡独立翻译，无视角色身份、对话结构和画面语境。CADGT 将漫画翻译建模为"微型话剧翻译"问题，五阶段流水线处理：

```
阶段1：页面理解
    ├─ 气泡文字定位（YOLOv8-OBB）
    ├─ 气泡→角色归属（尾巴检测 + 视觉特征）
    ├─ 场景分类（战斗/日常/恋爱/严肃）
    └─ 视觉特征提取（气泡颜色、形状、字体大小）

阶段2：对话图构建
    ├─ 日漫阅读顺序解析（右→左，上→下）
    ├─ 气泡按角色分组串联成"台词线"
    └─ 建立对话边关系（谁对谁说话 → 谁回应）

阶段3：角色画像
    ├─ 自称词提取（オレ/僕/わたし → 性格判定）
    ├─ 句尾习惯分析（だぜ/ですわ → 说话风格）
    ├─ 角色关系推断（敬语程度 → 社会关系）
    └─ 输出人设卡，随阅读进度动态更新

阶段4：上下文感知翻译
    ├─ 五维翻译上下文构建
    │   ├─ 场景维度（画面氛围）
    │   ├─ 角色维度（性格 + 说话风格）
    │   ├─ 历史维度（该角色此前说了什么）
    │   ├─ 对话维度（上下文的问答/回应关系）
    │   └─ 视觉维度（气泡区域截图）
    ├─ 简单气泡：OPUS-MT 快速直译（~50ms）
    └─ 复杂气泡：Qwen2.5-VL-2B 多模态上下文翻译（~3s）

阶段5：一致性校验
    ├─ 同角色台词语气一致性检测
    ├─ 对话衔接自然度检查
    ├─ 术语/人名翻译统一性校验
    └─ 发现不一致 → 回溯阶段4重译
```

**创新价值**：

| 对比维度 | 现有工具 | CADGT |
|------|:--:|:--:|
| 翻译单位 | 单气泡 | **整页对话图** |
| 角色感知 | 无 | **自动提取人设 + 语气适配** |
| 视觉上下文 | 不用 | **VLM 图文联合理解** |
| 场景适配 | 无 | **战斗/日常等场景感知** |
| 一致性保证 | 无 | **全页校验 + 回溯重译** |
| 速度优化 | 一律同等处理 | **两阶段混合调度（快译+精译）** |

---

### 创新五：图像修复 — LaMa 快速修复

**创新命名**：FFT全局感受野快速修复（LaMa-based Global Context Inpainting）

**核心思路**：采用基于傅里叶变换的 LaMa 模型，利用频域全局感受野实现大面积文字去除后的背景重建。

**创新价值**：传统 OpenCV 修复仅靠周围像素插值，无法处理网点纹理/速度线/渐变背景。LaMa 通过 FFT 在频域建模全局结构，对漫画特有的网点（halftone）纹理还原效果显著优于传统方法，模型仅 200MB，CPU 推理约 200ms/区域。

---

### 创新六：译文渲染 — 智能排版 + 风格匹配

**创新命名**：风格感知自适应排版渲染（Style-Aware Adaptive Typesetting, SAAT）

**核心思路**：系统的最终输出有两种形态——纯文本对照表（用于校对、学习、字幕导出）和气泡内替换渲染（原漫画阅读体验）。SAAT 专注后者，分五阶段，将 LaMa 修复后的干净漫画图像与 CADGT 译文结合，产出视觉上"像是原本就出版的中文版"的翻译漫画。

---

**阶段1：原文字风格检测**

在 LaMa 擦除文字**之前**，从原文字区域提取视觉风格参数。全部使用传统CV，无深度学习依赖。

| 检测项 | 方法 | 输出 |
|------|------|------|
| 文字主色调 | K-Means聚类（k=3）对文字区域像素做颜色量化，取占比最大的非背景色 | RGB颜色值 |
| 是否有描边 | Canny边缘检测 → 在文字轮廓外侧查找连续边缘带 | 描边宽度（px）+ 描边颜色 |
| 是否有阴影 | 文字区域与背景区域的亮度梯度分析，检测偏移的暗色块 | 阴影偏移量（dx, dy）+ 阴影颜色 |
| 字号估计 | 文字边界框高度 ÷ 字符行数，推算原文字号 | 估计字号（pt） |
| 字重判断 | 文字区域二值化后统计黑色像素占比 | 粗体/常规/细体 |
| 对齐方式 | 文字在气泡内的位置分析（左/中/右/两端） | 对齐模式 |

```
原文字区域截图（LaMa擦除前）
        │
        ├─→ 颜色聚类 → 主色=黑, 描边=白(2px)
        ├─→ 边缘检测 → 外侧有连续白边 → 有描边
        ├─→ 框高÷行数 → 字号≈14pt
        └─→ 二值化占比 → 字重=粗体

输出风格参数JSON:
{
  "text_color": "#000000",
  "stroke_color": "#FFFFFF",
  "stroke_width": 2,
  "has_shadow": false,
  "font_size": 14,
  "font_weight": "bold",
  "alignment": "center"
}
```

**阶段2：气泡可用空间计算**

LaMa 修复后，需要知道译文能放在哪里。可用空间不是简单的文本框矩形——需要还原到气泡的完整轮廓。

| 信息来源 | 获取方式 | 用途 |
|------|------|------|
| 原文字边界框 | 来自M2检测模块 | 核心排版区域的初始位置 |
| 气泡轮廓 | 从原图做颜色/边缘分割提取气泡区域 | 约束文字不能溢出气泡 |
| 气泡尾巴位置 | 来自M5阶段1的尾巴检测 | 排版时避开尾巴指向 |
| LaMa修复区域 | 修复蒙版范围 | 确认哪些像素可以覆盖文字 |

```
气泡轮廓 ┌──────────────┐
          │              │
          │  [可用区域]   │  ← 译文只能放在这个区域内
          │              │
          └──────┬───────┘
                 ↓ 尾巴（不能放文字）
```

**阶段3：自适应排版计算**

这是 SAAT 最核心的算法。

```
输入: 译文字符串 + 可用区域宽高 + 语言类型
        │
        ▼
┌──────────────────────────────────┐
│ Step 1: 确定排版方向              │
│                                  │
│ 原文竖排 + 语言为中文？            │
│   ├─ 是 → 转横排（中文竖排阅读差）  │
│   └─ 否（英文等）→ 横排            │
│ 原文横排 → 保持横排                │
└──────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────┐
│ Step 2: 二分搜索最佳字号           │
│                                  │
│ 字号范围: [8pt, 48pt]             │
│                                  │
│ while low ≤ high:                │
│   mid = (low+high)/2             │
│   用 mid 字号渲染译文             │
│   if 渲染结果 ≤ 可用区域:          │
│     记录 mid 为当前最佳           │
│     low = mid + 1  (试着更大)     │
│   else:                          │
│     high = mid - 1 (太大,缩小)    │
│                                  │
│ 输出: 最大可放入的字号             │
└──────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────┐
│ Step 3: 智能断行                  │
│                                  │
│ 中日文：按字断行，标点不置行首       │
│   例: "你在说什么啊"               │
│   一行放不下 → "你在说" / "什么啊"  │
│   调整: "你在" / "说什么啊" (语义)  │
│                                  │
│ 英文：按单词断行，连字符处理         │
│   例: "What are you"             │
│   一行放不下 → "What are" / "you" │
│                                  │
│ 混合（中英夹杂）: 中文按字、英文按词  │
└──────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────┐
│ Step 4: 行距与字距微调             │
│                                  │
│ 行数=1 → 行距=0（单行）           │
│ 行数=2 → 行距=字号×0.3            │
│ 行数≥3 → 行距=字号×0.2（紧凑）     │
│                                  │
│ 垂直居中计算：                     │
│   总渲染高度 = 行数×字号+(行数-1)×行距│
│   y偏移 = (可用区高度-总渲染高度)/2  │
└──────────────────────────────────┘
        │
        ▼
输出: 排版方案 {字号, 行数, 每行文本, 行距, 起始坐标, 对齐方式}
```

**排版自适应策略：**

| 场景 | 原文长度 | 译文长度 | 策略 |
|------|------|------|------|
| 日→中，译文短 | 何言ってんだよ（9字） | 你说什么啊（5字） | 字号放大，文字更清晰可读 |
| 日→中，译文适中 | 行くぞ（3字） | 上了（2字） | 字号微调，保持气泡视觉平衡 |
| 日→英，译文长 | 猫が好き（4字） | I like cats（11字符） | 适当缩小字号，换行压缩 |
| 中→英，译文极长 | 你好（2字） | Hello, how are you?（19字符） | 多行排版，最小可读字号兜底 |

**阶段4：风格渲染**

拿到排版方案后，在 LaMa 修复后的图像上绘制译文。

| 渲染步骤 | 实现 | 说明 |
|------|------|------|
| 阴影绘制（如有） | 先画偏移的暗色文字 | 在正式文字下层，根据阶段1的阴影偏移量绘制 |
| 描边绘制（如有） | 用描边颜色+字宽渲染 | PIL stroke宽度参数；Cairo text-stroke |
| 主体文字 | 用主色调+字号+字体 | 加载对应语种字体（中=思源黑体/日=Noto Sans JP/英=Roboto） |
| 竖排支持（极少用） | 逐字输出+逐行左移 | 当原文竖排且用户选择保留竖排时使用 |
| 叠加到原图 | 将渲染结果贴到LaMa修复后的区域 | PIL paste / Cairo 合成 |

**阶段5：两种输出形态**

SAAT 支持系统输出两种用户可选的结果：

**形态A：气泡内替换渲染（默认）**
```
输出：完整的漫画图像PNG
效果：看起来就像原本是中文出版的漫画
实现：LaMa修复 → SAAT排版渲染 → 叠加合成
用途：直接阅读、分享、发布
```

**形态B：纯文本对照表**
```
┌────────────────────────────────────┐
│  角色A（热血男主・自称オレ）         │
│  原文: 行くぞ！                     │
│  译文: 上了！                       │
│  ───────────────────               │
│  角色B（高冷对手）                  │
│  原文: お前の力はそんなものか         │
│  译文: 你的实力就这点程度吗          │
│  ───────────────────               │
│  角色A                             │
│  原文: まだまだこれからだ！           │
│  译文: 还没完呢！                   │
└────────────────────────────────────┘

输出：文本文件或GUI面板中显示
用途：校对翻译、学习日语、字幕导出、替代阅读方式
```

**创新价值总结**：

| 对比维度 | 现有工具 | SAAT |
|------|------|------|
| 字号 | 固定或手动调节 | **自适应气泡空间，自动计算最优字号** |
| 排版方向 | 全部横排 | **检测原文方向，智能选择横/竖排** |

## 各技术方案详细对比论证

---

### 一、文本检测：为何选 YOLOv8-OBB + DBNet

| 对比维度 | YOLOv8（原始） | RT-DETR | CRAFT | **YOLOv8-OBB + DBNet** |
|------|:--:|:--:|:--:|:--:|
| 竖排文本适配 | ❌ 水平框 | ⚠️ 仍为水平框 | ⚠️ 字符级框 | **✅ 旋转框 + 分割** |
| 弧形文本 | ❌ | ⚠️ | ✅ | **✅ DBNet 分割** |
| 推理速度 | 极快 | 快 | 慢 | **快（OBB）+ 快（DBNet）** |
| 训练数据需求 | 中 | 高 | 中 | **中** |
| 社区生态 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ | **⭐⭐⭐⭐（YOLO）+ ⭐⭐⭐⭐（PaddleOCR）** |
| 落地难度 | 低 | 中高 | 中 | **低** |

**选择理由**：

1. **YOLOv8-OBB 解决竖排**：日漫约30-40%文本为竖向排版，水平矩形框会框入大量非文字区域，影响后续OCR和修复精度。旋转框天然适配竖直文本，标注成本仅增加角度一个维度，但效果提升显著。

2. **DBNet 解决不规则**：拟声词（ドドド、ゴゴゴ）往往跟随画面元素弯曲变形，旋转矩形框也无法精确包裹。DBNet 以分割方式输出任意形状文本区域，作为OBB的补充通道。

3. **生态继承**：OBB 共享 YOLOv8 全部预训练权重和工具链，DBNet 是 PaddleOCR 内置模块。两个模型都无需从零训练，微调即可。

4. **为什么不选 RT-DETR/CRAFT**：RT-DETR 虽无需NMS但训练资源需求大，不适合大学生计算条件；CRAFT 字符级检测精度高但推理慢，不适合实时交互式工具。

---

### 二、图像增强：为何选 TACE 三级级联

| 对比维度 | 全图超分（A） | 纯数据增强（B） | **TACE 三级级联（C）** |
|------|:--:|:--:|:--:|
| 轻度模糊处理 | ✅ 但浪费 | ✅ | **✅ CLAHE 零成本** |
| 中度模糊处理 | ✅ | ⚠️ 仍需样本覆盖 | **✅ SVTR 抗模糊训练** |
| 重度模糊处理 | ✅ | ❌ 无力 | **✅ Real-ESRGAN fallback** |
| 推理耗时（正常图） | +5-10s | 0 | **0（一级即过）** |
| 推理耗时（模糊图） | +5-10s | 0 | **+100ms/框（仅fallback触发）** |
| 伪影风险 | 有 | 无 | **仅重度模糊时有** |
| 额外模型维护 | +1模型 | 0 | **+1模型（仅fallback用）** |

**选择理由**：

1. **成本与收益的帕累托最优**：漫画中70%+的文字区域是清晰的（出版级印刷/扫描），全图超分对70%情况纯属浪费。TACE 对清晰文本零开销，对模糊文本精准投入。

2. **CLAHE 的免费午餐**：对比度受限自适应直方图均衡化（CLAHE）是 OpenCV 内置函数，一行代码，约0.1ms执行时间。对轻度模糊（如扫描阴影导致的局部对比度下降）效果立竿见影。

3. **SVTR 的天生抗噪**：Transformer 的自注意力机制比 CNN+RNN 对局部噪声更鲁棒，训练时混合模糊增强样本，可覆盖中度模糊场景。

4. **Real-ESRGAN 只做兜底**：极度模糊（如低分辨率网络截图）才触发，且只对几十像素宽的文本框做超分，耗时可控。

### 三、OCR识别：为何 SVTR 替代 CRNN

| 对比维度 | CRNN + Attention（原始） | EasyOCR | TrOCR | **SVTR** |
|------|:--:|:--:|:--:|:--:|
| 架构 | CNN → RNN 串行 | CNN → LSTM 串行 | ViT → Transformer | **全Transformer并行** |
| 字符建模方式 | 逐片感受野，局部特征 | 同CRNN | 全局自注意力 | **混合注意力（局部+全局+池化）** |
| 中文精度 | ~93% | ~87% | ~95% | **~96%** |
| 日文精度 | ~91% | ~80% | ~93% | **~94%** |
| 变形字体鲁棒性 | 弱（CNN局部视野受限） | 弱 | 强 | **强（全局结构建模）** |
| 推理速度（基准） | 1x | 0.8x | 0.4x | **1.3x** |
| 训练速度（基准） | 1x | — | 0.5x | **1.5x** |
| 模型大小 | ~8MB | ~15MB | ~300MB+ | **~5MB（Tiny）** |
| 竖排文本 | 需旋转校正 | 几乎不支持 | 需旋转校正 | **内置方向分类器** |

**选择理由**：

1. **架构代际优势**：CRNN 是2015年方案（CNN+RNN+CTC），SVTR 是2022年方案（纯Transformer）。前者串行，后者并行，在精度和速度两个维度上同时超越。

2. **漫画场景的关键差异**：漫画文字的变形（透视、拉伸、艺术化）本质是对字符"整体结构"的破坏。CRNN 从左到右看局部片段，某个片段变形就会出错。SVTR 的全局自注意力"一眼看全貌"，即使局部变形，整体字形结构仍可辨识。

3. **微调兼容性**：PP-OCRv5 已将 SVTR 作为官方推荐配置，微调工具链、预训练权重、导出脚本全部现成。从 CRNN 切换到 SVTR 仅需修改配置文件中的一个参数名。

4. **轻量可部署**：SVTR-Tiny 仅 5MB，CPU 上毫秒级推理，适合 QT 桌面端集成。

5. **不选 EasyOCR 的原因**：日文识别精度显著不足（~80%），竖排文本几乎没有支持，虽然 PyTorch 生态统一但性能牺牲太大。

6. **不选 TrOCR 的原因**：精度虽高但模型过大（300MB+），CPU 推理太慢，不符合桌面端实时交互需求。作为学生项目，大模型的训练和调试成本过高。

### 四、机器翻译：为何 CADGT 替代传统 NMT

| 对比维度 | 传统NMT逐句（原始） | 纯SLM逐气泡 | CADGT（本方案） |
|------|:--:|:--:|:--:|
| 翻译单位 | 单气泡 | 单气泡 | **整页对话图** |
| 角色感知 | ❌ | 部分（prompt描述） | **✅ 自动提取+动态人设卡** |
| 对话上下文 | ❌ | ⚠️ 手动拼接 | **✅ 结构化对话图驱动** |
| 视觉上下文 | ❌ | ❌ | **✅ VLM 图文联合理解** |
| 场景适配 | ❌ | 部分 | **✅ 战斗/日常等场景分类** |
| 翻译一致性 | ❌ 无保证 | ❌ 无保证 | **✅ 全页校验+回溯重译** |
| 逐句精度 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 对话连贯性 | ⭐ | ⭐⭐⭐ | **⭐⭐⭐⭐⭐** |
| 速度（正常页） | <1s | 10-15s | **6-7s（两阶段混合）** |
| 离线可用 | ✅ | ✅ | ✅ |
| 学术创新度 | 低 | 中 | **高** |

**选择理由**：

1. **彻底改变翻译范型**：现有所有翻译工具（包括商业的 Google Translate 图片翻译、开源的 BallonTranslator、manga-image-translator）都遵循"检测→识别→逐句翻译→渲染"流程。CADGT 跳出了这个框架，让模型在翻译之前先"理解"整页漫画的对话结构。

2. **对话图的价值**：漫画不是孤立句子的集合，而是角色间的对话。同一角色连续说话时存在语气延续，不同角色对话时存在因果回应。传统逐句翻译割裂了这些关系，CADGT 通过对话图显式建模这些关系。

3. **角色画像是可持续累积的信息**：首次阅读一部漫画时建立的角色画像，可以被同一部漫画的后续所有页面复用。读得越多，翻译得越准——这是一个传统方案不具备的"持续学习"特性。

4. **五维上下文的组合优势**：单靠文字、单靠视觉、单靠角色信息都不足以做出最佳的翻译选择，五维信息的交叉验证才能让模型在歧义场景中做出正确的风格选择。

5. **两阶段混合调度兼顾速度和质量**：60%的简单气泡由 OPUS-MT 在毫秒级完成，用户感知到的"首次翻译结果"几乎即时；40%的复杂气泡由 VLM 深度处理，作为精修层叠加，最终结果在6-7秒内交付。

6. **不选纯NMT的原因**：OPUS-MT 等传统 NMT 是优秀的逐句翻译器，但完全不具备上下文理解能力。同一角色三句话，可能译出三种语气。

7. **不选纯SLM的原因**：纯大模型逐气泡翻译虽然质量可以，但缺乏结构化理解。模型"知道"自己在翻译一个漫画气泡，但不知道这个气泡在一页漫画里的位置和角色。CADGT 的结构化管线补上了这些信息。

### 五、图像修复：为何 LaMa 替代传统 OpenCV inpainting

| 对比维度 | OpenCV Telea/NS（常见方案） | AOT-GAN | **LaMa** |
|------|:--:|:--:|:--:|
| 纯色背景填充 | ✅ | ✅ | ✅ |
| 网点纹理（halftone） | ❌ 模糊一团 | ✅ 还原好 | **✅ 还原好** |
| 速度线/渐变背景 | ❌ 明显断裂 | ✅ | **✅** |
| 大区域填充 | ❌ 模糊扩散 | ⚠️ 可能不稳定 | **✅ FFT全局建模** |
| 推理速度 | <1ms | ~1-2s/区域 | **~200ms/区域** |
| 模型大小 | 0（传统算法） | ~500MB | **~200MB** |
| 落地难度 | 零 | 高 | **低（HuggingFace即用）** |

**选择理由**：

1. **FFT 是漫画修复的天然优势**：漫画背景大量使用网点纸（halftone），它本质是周期性的点阵图案。LaMa 在频域（傅里叶变换）做修复，对周期性纹理的感知和处理天然强于在空间域做修复的方法。

2. **大蒙版友好**：检测阶段输出的文字区域框通常需要适当膨胀以确保文字完全被覆盖，这意味着修复蒙版往往比实际文字大。LaMa 设计初衷就是处理大面积缺失，比 Telea 的"从边缘向内扩散"策略可靠得多。

3. **可控的资源消耗**：200MB 模型 + 200ms/区域，在一页10个气泡的情况下约2秒。用户感知可以接受，且可以批量并行处理多个气泡区域。

4. **不选 OpenCV 的原因**：Telea 和 NS 算法本质是加权扩散，对简单纯色背景有效，但漫画中大量的网点、速度线、渐变背景会暴露其局限。当背景有结构时，简单的像素插值无法重建结构，导致填充区域"糊"成一片。

### 六、译文渲染：为何 SAAT 替代简单填充

| 对比维度 | 固定字体填充 | 自适应排版 | **SAAT 风格感知排版** |
|------|:--:|:--:|:--:|
| 字体大小 | 固定 | 自适应气泡尺寸 | **自适应 + 原文字号参照 + 二分搜索最优** |
| 换行 | 机械按字符数断行 | 按字/词智能断行 | **语义级断行（标点规则+词边界+混合语种）** |
| 排版方向 | 全部横排 | 全部横排 | **检测原文方向，横/竖排智能选择** |
| 文字颜色 | 固定（黑/白） | 固定 | **K-Means聚类检测原文颜色并匹配** |
| 描边/阴影 | 无 | 无 | **Canny边缘检测发现描边/阴影并复现** |
| 气泡空间约束 | 不管 | 简单矩形约束 | **气泡轮廓提取，精确约束文字不溢出** |
| 输出形态 | 仅渲染图 | 仅渲染图 | **渲染图 + 角色对照文本表双输出** |
| 竖排转横排 | 不支持 | 不支持 | **原文竖排→译文智能转横排（中文竖排阅读体验差）** |
| 渲染效果 | "贴上去的翻译" | "刚好塞进去的" | **"这是这部漫画本来的样子"** |

**选择理由**：

1. **渲染是用户体验的最终出口**：检测、OCR、翻译做得再好，如果渲染出的文字字号不对、颜色突兀、排版混乱，用户的第一感知就是"这工具不行"。SAAT 将渲染从"后处理"提升为独立的技术模块。

2. **风格匹配决定真实感**：漫画中文字的视觉风格是信息的一部分——加粗=吼叫、小字=耳语、红字=愤怒、描边=强调。如果不复现这些风格，翻译后的漫画会丢失情绪信息。SAAT 通过纯CV检测原文风格并复现，不增加模型依赖。

3. **关键的技术创新是二分搜索自适应字号**：传统方案要么固定字号（译文长短不定时溢出或留白），要么手动调节（打断用户流程）。SAAT 的二分搜索在 8pt-48pt 范围内自动找到最大可放入的字号，保证译文清晰可读，同时不溢出气泡。

4. **气泡轮廓约束比矩形约束更精准**：漫画气泡是不规则形状（椭圆形、爆炸形、云朵形），仅用检测到的文字矩形框作为渲染边界会浪费气泡空间或溢出到非文字区域。气泡轮廓分割是从原图提取的额外约束，让排版更精确。

5. **双输出形态覆盖不同使用场景**：
   - 渲染图：直接阅读翻译后的漫画
   - 文本对照表：学习日语、校对翻译质量、导出字幕
   - 两个形态共享同一套CADGT翻译结果，无额外计算成本

6. **不选简单填充和普通自适应排版的原因**：前者完全忽视漫画的视觉设计，后者只解决了"字能放进去"的问题，但没解决"放进去之后看起来对不对"。SAAT 追求的是翻译后的漫画在视觉上不违和——读者注意力应该放在内容理解上，而不是被"这段文字看起来不对劲"所分散。

7. **全部传统CV实现，零模型依赖**：风格检测（K-Means颜色聚类、Canny边缘检测）和排版计算都是纯算法，不需要GPU推理，毫秒级完成。这是 SAAT 在精度和效率之间的最优平衡。

### 七、GUI：为何 PySide6

| 对比维度 | QT C++ | QT PySide6 |
|------|:--:|:--:|
| 与算法管线对接 | 需跨语言（C++↔Python） | **零跨语言开销** |
| 调试效率 | 低（编译+链接） | **高（解释执行）** |
| 开发速度 | 慢 | **快（2-3倍）** |
| 性能 | 最优 | **够用（GUI非瓶颈）** |
| 部署 | 单exe | **pyinstaller → 单exe** |
| 学习曲线 | 陡（C++ + QT） | **平（Python + QT）** |
| 社区教程 | 多 | **多（近年增长迅速）** |

**选择理由**：整个管线（检测、识别、翻译、修复、渲染）均为 Python 生态。选择 C++ QT 意味着每个模块的每次调用都需要跨语言桥接，开发和调试成本剧增。PySide6 在 Python 层面统一调用所有模块，开发效率是 C++ 方案的2-3倍。GUI 本身不是性能瓶颈——耗时在算法管线，不在界面渲染。

## 模块划分

系统划分为 **4层、9个模块**：

### 第一层：数据层

**Module 1: 模型管理器 (ModelManager)**

| 职责 | 说明 |
|------|------|
| 模型加载/卸载 | 懒加载策略，用时才加载到内存 |
| 模型版本管理 | 微调版本的追踪与切换 |
| ONNX导出与缓存 | 统一管理各模型的ONNX导出 |
| 角色画像持久化 | 一部漫画的人设卡存为JSON，同作品后续复用 |

### 第二层：算法管线层

**Module 2: 检测模块 (Detector)** — YOLOv8-OBB + DBNet

| 职责 | 说明 |
|------|------|
| YOLOv8-OBB 推理 | 主力检测，输出旋转矩形框 |
| DBNet 推理 | 辅助检测，输出任意形状文本区域 |
| 双路结果融合 | NMS去重 + 结果合并策略 |
| 气泡尾巴检测 | 识别气泡尖角指向，判断归属角色 |
| 输出 | `[TextBox{坐标, 旋转角, 归属角色, 视觉特征}]` |

**Module 3: 增强模块 (Enhancer)** — TACE

| 职责 | 说明 |
|------|------|
| 一级增强 | CLAHE对比度 + Unsharp Mask锐化（传统CV，<1ms） |
| 二级透传 | 直接交付SVTR识别（利用原生抗模糊能力） |
| 三级超分 | Real-ESRGAN定向超分（仅置信度不足时触发） |
| 输出 | 增强后的文本区域图像 |

**Module 4: 识别模块 (Recognizer)** — PP-OCRv5 + SVTR

| 职责 | 说明 |
|------|------|
| 文本方向检测 | 竖排/倒置文本自动旋转校正 |
| SVTR推理 | 中日英多语种文本识别 |
| 置信度反馈 | 低置信度 → 触发Enhancer三级超分 → 重试 |
| 输出 | 文本字符串 + 语种标签 + 置信度分数 |

**Module 5: 翻译模块 (Translator)** — CADGT 核心（最大最复杂模块）

| 子模块 | 职责 | 说明 |
|------|------|------|
| 页面理解 | 场景分类 + 阅读顺序解析 | 战斗/日常/恋爱/严肃；日漫右→左、上→下 |
| 对话图构建 | 气泡→角色分组 + 对话边关系 | 核心数据结构：`Graph(节点=气泡, 边=顺序+应答)` |
| 角色画像 | 自称词/句尾词提取 → 人设卡 | オレ→热血, わたし→文静; 动态更新 |
| 翻译引擎 | 五维上下文 + 两阶段混合调度 | 简单气泡OPUS-MT(~50ms)；复杂气泡Qwen2.5-VL-2B(~3s) |
| 一致性校验 | 语气/衔接/术语检查 → 回溯重译 | 整页翻译完成后统一检查 |
| 输出 | 结构化译文列表 | `[{气泡ID, 原文, 译文, 置信度, 翻译方式}]` |

**Module 6: 修复模块 (Inpainter)** — LaMa

| 职责 | 说明 |
|------|------|
| 蒙版生成 | 膨胀文本框区域确保完全覆盖原文字 |
| LaMa推理 | FFT全局感受野背景重建 |
| 输出 | 去除原文字后的干净图像 |

**Module 7: 渲染模块 (Renderer)** — SAAT

渲染模块是系统的用户可见终点，决定翻译后的漫画看起来是否自然。分五个子模块：

**子模块7.1：风格检测 (StyleDetector)**

所有分析基于 LaMa 擦除前的原文字区域，全部使用传统CV。

| 检测项 | 方法 | 输出值 | 用途 |
|------|------|------|------|
| 文字主色 | K-Means (k=3) 像素聚类，排除背景色后取最大簇中心 | RGB值 | 确定译文字体颜色 |
| 描边检测 | Canny边缘检测 → 在文字轮廓外侧N像素内搜索连续边缘带 | 描边宽度(px) + 描边颜色 | 复现描边效果 |
| 阴影检测 | 文字区域与背景的亮度梯度分析，检测偏移暗色块 | 偏移量(dx,dy) + 阴影颜色 | 复现阴影效果 |
| 字号估计 | 文字框高度 ÷ 行数（行数通过水平投影波谷计数） | 估计字号(pt) | 作为二分搜索的参考起点 |
| 字重判断 | 二值化后黑色像素占比 → 映射到粗体/常规/细体 | 字重分类 | 选择对应粗细的字体文件 |
| 对齐方式 | 文字在气泡框内的水平位置分布分析 | 左/中/右 | 确定译文对齐方式 |

**子模块7.2：空间计算 (SpaceCalculator)**

| 输入 | 来源 | 用途 |
|------|------|------|
| 原文字边界框 | M2 检测模块 | 初始排版区域位置 |
| 气泡轮廓蒙版 | 原图颜色/边缘分割 | 约束文字不溢出气泡的不规则边界 |
| 气泡尾巴位置 | M5 阶段1 尾巴检测结果 | 排版时避开尾巴区域 |
| LaMa修复区域 | M6 修复蒙版 | 确认可覆盖的像素范围 |

**子模块7.3：自适应排版 (TypeSetter)** — SAAT 核心算法

```
输入: 译文字符串 + 可用区域(宽,高) + 语言类型 + 输入来源风格参数
        │
        ▼
Step 1: 确定排版方向
  原文竖排 → 中文/英文 → 转横排（中文竖排阅读体验差）
  原文横排 → 保持横排
  
Step 2: 二分搜索最佳字号
  范围: [8pt, 48pt]
  条件: 渲染结果宽≤可用区宽 AND 渲染结果高≤可用区高
  目标: 最大化字号（而不是刚好放入）
  
Step 3: 智能断行
  中日文: 按字断行，标点不能置行首（中文排版规则）
  英文:   按词断行，必要时加连字符
  混合:   中文按字、英文按词
  
  示例（可用区窄）:
  "你在说什么啊" → ["你在", "说什么啊"]  (避免"你在说"/"什么啊"的语义割裂)

Step 4: 行距/字距微调
  行数=1: 行距=0
  行数=2: 行距=字号×0.3
  行数≥3: 行距=字号×0.2
  
  垂直居中: y_offset = (可用区高 - 总渲染高) / 2
  水平对齐: 跟随阶段1检测到的原文对齐方式
```

| 排版场景 | 策略 |
|------|------|
| 译文比原文短（日→中常见） | 字号放大，文字更清晰可读，充分利用气泡空间 |
| 译文比原文长（日→英常见） | 适当缩小字号+多行排版，最小字号10pt兜底保证可读 |
| 气泡空间紧张 | 字号降到可读底线(10pt) + 最小行距，保证完整显示 |
| 气泡空间充裕 | 字号放大到视觉舒适为止，保持漫画阅读体验 |
| 竖排原文 | 默认转横排输出中文，极少数艺术场景保留竖排 |

**子模块7.4：风格渲染 (Painter)**

按层次顺序绘制，确保视觉效果正确：

| 层级 | 绘制内容 | 实现方式 | 说明 |
|:--:|------|------|------|
| 1 | 阴影 | PIL: 先画偏移暗色文字；Cairo: 设置shadow参数 | 仅当风格检测到原文有阴影时执行 |
| 2 | 描边 | PIL: `stroke_width` + `stroke_fill`；Cairo: `text-stroke` | 用阶段1检测到的描边颜色和宽度 |
| 3 | 主体文字 | 加载语种对应字体（中=思源黑体, 日=Noto Sans JP, 英=Roboto） | 使用排版模块输出的字号/行距/坐标 |
| 4 | 叠加原图 | PIL `Image.paste()` / Cairo `paint()` | 渲染结果覆盖到LaMa修复后的干净图像上 |

**子模块7.5：双输出 (Exporter)**

| 输出形态 | 内容 | 格式 | 用途 |
|------|------|------|------|
| **渲染图** | LaMa修复 + SAAT渲染后的完整漫画页面 | PNG | 直接阅读翻译后的漫画 |
| **文本对照表** | 按角色分组的原文→译文对照，含场景和人设信息 | TXT / GUI面板 | 校对翻译、学习日语、导出字幕 |

两种输出共享同一份CADGT翻译结果，文本对照表为渲染图免费附带，无额外计算成本。

## 可行性评估

### 一、技术可行性

| 模块 | 核心技术 | 成熟度 | 开源实现 | 风险 |
|------|------|:--:|:--:|:--:|

| M3 增强 | CLAHE + Real-ESRGAN | ✅ 成熟 | OpenCV / Real-ESRGAN官方 | 🟢 低 |
| M4 OCR | PP-OCRv5 + SVTR | ✅ 成熟 | PaddleOCR官方 | 🟢 低 |
